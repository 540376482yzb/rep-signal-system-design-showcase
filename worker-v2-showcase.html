The Robustness Upgrade

Child 23 simplified the contracts. This follow-up keeps that package boundary, then restores durable orchestration from child 20 with a queue library instead of hand-rolled polling.

1

Internalize Scheduling

GitHub Action calls one internal endpoint. The API owns due-profile selection, cadence windows, market-row expansion, schedule-run creation, and enqueue behavior.

2

Queue Execution

BullMQ owns delivery mechanics: waiting, active, delayed, retry, backoff, concurrency, failure events, and rate limits around expensive curation work.

3

Persist Business Truth

Postgres remains the source for run status, job progress, candidate signals, email sends, dedupe history, and reconciliation decisions.

Retire From Child 23

Do not make GHA loop every market row, hold candidates in workflow memory, select the lead, or call send endpoints row by row.

Keep From Child 23

Use getRepSignalStrataPackage as the clean raw-data boundary for menu, mention, and sales curation.

Postgres State Model

BullMQ is the queue, not the business ledger. Postgres answers what happened to each profile run after workers, retries, and Redis retention have moved on.

rep_signal_profiles

Profile, delivery settings, cadence, and configured channel/market keys.

profile_id email market_keys channel_keys feed_config_json

rep_signal_schedule_runs

One orchestration record per profile and cadence window.

rep_signal_job_runs

Thin ledger for BullMQ jobs. Not a queue table.

rep_signal_candidate_signals

Send intent, provider result, dedupe history, and reconciliation state.

Runtime Flow

The API creates the run and jobs. Workers process market rows through the package-era extractor. Finalization waits on Postgres terminal state, not Redis internals.

Create run, expand market rows, create job ledger rows, enqueue deterministic BullMQ jobs.

Failure And Recovery

The robustness story is not "the queue fixes everything." BullMQ handles delivery mechanics; Postgres constraints and idempotent workers protect business correctness.

Schedule / Enqueue Repeated triggers must not duplicate a cadence-window run. If ledger creation and enqueue cannot complete safely, the API fails before reporting accepted.
Market Row Curation Transient failures retry with bounded backoff. Exhausted jobs become dead in the ledger, and finalization can continue once all rows are terminal.
Email Send A sending row is written before provider calls. Stale sending rows block duplicate sends until reconciliation marks the outcome.
